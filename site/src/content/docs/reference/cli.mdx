## Overview

```
kinoko - Knowledge sharing infrastructure for AI agents

USAGE:
  kinoko [command] [flags]

AVAILABLE COMMANDS:
  init      Initialize Kinoko workspace
  serve     Start the Kinoko infrastructure server
  run       Start the local agent daemon
  extract   Run extraction pipeline on a session log file
  import    Import session logs into the extraction queue
  ingest    Import a markdown file as a skill through the quality critic
  match     Find skills matching a query
  scan      Scan files for credentials and secrets
  index     Index a skill repo into SQLite
  pull      Clone or update skill repositories
  queue     Queue inspection and management
  rebuild   Rebuild SQLite cache from git repos
  stats     Print client pipeline metrics
  help      Help about any command

GLOBAL FLAGS:
  -h, --help      Help for kinoko
      --version   Version for kinoko
```

## `kinoko init`

Initialize a new Kinoko workspace in `~/.kinoko/`.

```bash
kinoko init [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--connect` | string | `""` | Connect to a Kinoko server URL (e.g. `localhost:23231`) |
| `--token` | string | `""` | Registration token for server authentication (used with `--connect`) |

**Creates:**
- `~/.kinoko/` — workspace root
- `~/.kinoko/config.yaml` — default configuration
- `~/.kinoko/cache/` — local skill cache directory
- `~/.kinoko/id_ed25519` — client SSH key for server authentication

**Behavior:**
- Safe to run multiple times — won't overwrite existing files
- With `--connect`, configures the workspace as a client of the given server and auto-registers the client SSH key
- With `--connect` and `--token`, includes the token as a Bearer credential when registering the SSH key (required if the server has `server.registrationToken` configured)
- Without `--connect`, sets up for solo use (localhost)
- Interactive LLM setup wizard detects existing credentials and guides you through provider selection

## `kinoko doctor`

Validate your Kinoko installation by checking configuration, database access, SSH keys, server connectivity, and LLM credentials.

```bash
kinoko doctor [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--config` | string | `~/.kinoko/config.yaml` | Path to config file |

**Checks performed:**

1. **Config file** — exists and parses correctly
2. **Queue database** — path is writable for SQLite operations
3. **Cache directory** — exists for local skill storage
4. **SSH key** — exists for git server authentication
5. **Server** — reachable on configured port (localhost:23233 by default)
6. **LLM credentials** — resolvable via the 10-step credential chain
7. **LLM connectivity** — actual API test with latency measurement

**Exit codes:**

| Code | Meaning |
|------|---------|
| 0 | All checks passed |
| 1 | Issues found (see output for details) |

**Example output:**
```
🍄 Kinoko Doctor

  ✓ Config file      /home/user/.kinoko/config.yaml
  ✓ Queue database   /home/user/.kinoko/queue.db
  ✓ Cache directory  /home/user/.kinoko/cache/
  ✓ SSH key          /home/user/.kinoko/id_ed25519
  ✗ Server           localhost:23233 (connection refused)
  ✓ LLM credentials  Anthropic (setup token)
  ✓ LLM connectivity claude-opus-4-0-20250514 (247ms)

  1 issue found. Run 'kinoko serve' to start the server.
```

**Common issues:**
- Server not running: start with `kinoko serve`
- Missing LLM credentials: run `kinoko init` for the setup wizard, or set environment variables

## `kinoko serve`

Start the shared Kinoko infrastructure server.

```bash
kinoko serve [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--config` | string | `~/.kinoko/config.yaml` | Path to config file |

**What it starts:**
- **Soft Serve git server** (SSH) — skill repos live here
- **Discovery API** (HTTP) — `/api/v1/discover`, `/api/v1/health`, `/api/v1/ingest`, `/api/v1/register`
- **Git hooks** — credential scanning (pre-receive) and auto-indexing (post-receive)
- **SQLite indexer** — derived cache from git repos

**Behavior:**
- Self-bootstraps on first run: creates data dir and admin keypair
- In solo mode, auto-registers the local client SSH key (`~/.kinoko/id_ed25519.pub`) so no manual key setup is needed
- Binds to `server.host`:`server.port` from config (default `127.0.0.1:23231`)
- Handles SIGINT/SIGTERM for graceful shutdown
- This is **pure infrastructure** — run `kinoko run` separately for the agent daemon

**Environment variables:**

| Variable | Purpose |
|---|---|
| `OPENAI_API_KEY` | Fallback API key for embeddings (used by indexer) |
| `KINOKO_EMBEDDING_API_KEY` | API key for embeddings (overrides `OPENAI_API_KEY`) |

**Example:**
```bash
kinoko serve
kinoko serve --config ./my-config.yaml
```

## `kinoko run`

Start the local agent daemon that runs alongside your AI agents.

```bash
kinoko run [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--config` | string | `~/.kinoko/config.yaml` | Path to config file |
| `--server` | string | from config | Server URL override (e.g. `localhost:23231`) |

**What it starts:**
- **Worker pool** — extracts knowledge from local session logs
- **Scheduler** — decay cron, stale sweep, periodic stats
- **Injection** — loads relevant skills into agent sessions

**Behavior:**
- Reads server URL from config to push extracted skills
- **Degraded mode:** works without LLM key — scheduler and injection still run, only extraction is disabled
- Without embedding key, injection is disabled
- Handles SIGINT/SIGTERM for graceful shutdown

**Environment variables:**

| Variable | Purpose |
|---|---|
| `OPENAI_API_KEY` | Fallback API key for both embeddings and LLM |
| `KINOKO_EMBEDDING_API_KEY` | API key for embeddings (overrides `OPENAI_API_KEY`) |
| `KINOKO_LLM_API_KEY` | API key for LLM calls (overrides `OPENAI_API_KEY`) |

**Example:**
```bash
export OPENAI_API_KEY=sk-...
kinoko run
kinoko run --server my-server:23231
```

## `kinoko extract`

Run the 3-stage extraction pipeline on a session log file. Parses the session, runs Stage 1 (filter) → Stage 2 (scorer) → Stage 3 (critic), performs a novelty check against existing skills, and pushes the result to the skill library via git.

```bash
kinoko extract <session-log> [flags]
```

**Arguments:**

| Argument | Description |
|---|---|
| `<session-log>` | Path to the session log file |

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--config` | string | `""` | Config file path |
| `--library` | string | first configured | Library ID to use |
| `--api-url` | string | `$KINOKO_API_URL` or `http://127.0.0.1:23233` | Kinoko API URL for embedding and novelty checks |
| `--dry-run` | bool | `false` | Run the full pipeline but skip git push |
| `--timeout` | duration | `5m` | Command timeout |

**Pipeline flow:**

1. **Parse** — reads the session log and extracts metadata
2. **Stage 1 (Filter)** — fast heuristic check, rejects trivial sessions
3. **Stage 2 (Scorer) — LLM rubric scoring for quality assessment
4. **Stage 3 (Critic)** — LLM evaluates the candidate skill (substitution test, hard reject triggers, SKILL.md generation)
5. **Novelty check** — queries the server to ensure the skill isn't a duplicate
6. **Git push** — commits the skill to the library (skipped with `--dry-run`)

**Environment variables:**

| Variable | Purpose |
|---|---|
| `OPENAI_API_KEY` | Fallback API key for embeddings and LLM |
| `KINOKO_EMBEDDING_API_KEY` | API key for embeddings (overrides `OPENAI_API_KEY`) |
| `KINOKO_LLM_API_KEY` | API key for LLM calls (required; overrides `OPENAI_API_KEY`) |

**Examples:**
```bash
export OPENAI_API_KEY=sk-...
kinoko extract ./session.log
kinoko extract ./session.log --library my-team --config ./config.yaml
kinoko extract ./session.log --dry-run          # evaluate without pushing
kinoko extract ./session.log --timeout 10m      # longer timeout for large sessions
kinoko extract ./session.log --api-url http://kinoko.internal:23233
```

**Output:** Prints a human-readable summary followed by JSON `ExtractionResult` including stage results, skill record (if extracted), commit hash, timing, and status.

**Exit codes:**

| Code | Meaning |
|------|---------|
| 0 | Skill extracted successfully |
| 2 | Extraction rejected (session didn't pass quality gates) |
| 3 | Extraction error |

## `kinoko ingest`

Import a markdown file as a skill through the Stage 3 LLM critic. The critic applies the same quality gate as extraction: substitution test, hard reject triggers, and SKILL.md generation. Raw input is transformed into a well-structured skill.

```bash
kinoko ingest <file.md> [flags]
```

**Arguments:**

| Argument | Description |
|---|---|
| `<file.md>` | Path to the markdown file to ingest |

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--name` | string | derived from filename | Skill name (kebab-case) |
| `--category` | string | from LLM or `tactical` | Skill category (overrides LLM/front matter) |
| `--tags` | string | `""` | Comma-separated tags (overrides LLM/front matter) |
| `--library` | string | `local` | Library ID |
| `--force` | bool | `false` | Skip critic — validate structure and push as-is (requires valid YAML front matter with `name` field) |
| `--api-url` | string | `$KINOKO_API_URL` or `http://127.0.0.1:23233` | Kinoko API URL |
| `--dry-run` | bool | `false` | Evaluate only, don't push to git |
| `--timeout` | duration | `5m` | Command timeout |
| `--config` | string | `""` | Config file path |

**Flow (normal mode):**

1. **Read** — loads the markdown file
2. **Stage 3 Critic** — LLM evaluates quality, generates structured SKILL.md
3. **Metadata extraction** — parses name, category, tags from generated front matter (CLI flags override)
4. **Embedding** — computes vector via server API
5. **Novelty check** — rejects duplicates
6. **Index** — stores in local SQLite
7. **Git push** — commits to library (skipped with `--dry-run` or if duplicate)

**Flow (`--force` mode):**

1. **Read** — loads the markdown file
2. **Validate** — checks UTF-8, size (≤1 MB), and valid YAML front matter with `name` field
3. **Index + Push** — skips critic, indexes and pushes directly

**Environment variables:**

| Variable | Purpose |
|---|---|
| `OPENAI_API_KEY` | Fallback API key for LLM |
| `KINOKO_LLM_API_KEY` | API key for LLM calls (overrides `OPENAI_API_KEY`) |
| `ANTHROPIC_API_KEY` | API key for Anthropic models (overrides `OPENAI_API_KEY`) |

**Examples:**
```bash
kinoko ingest CLAUDE.md
kinoko ingest my-notes.md --category LEARN --tags go,testing
kinoko ingest SKILL.md --force              # skip critic, validate and push as-is
kinoko ingest draft.md --dry-run            # evaluate without pushing
kinoko ingest doc.md --name my-skill --library team-alpha
```

**Exit codes:**

| Code | Meaning |
|------|---------|
| 0 | Skill ingested successfully |
| 2 | Rejected by critic |

## `kinoko import`

Parse session log files and enqueue them for extraction. Does **not** start workers — sessions wait for `kinoko run` to process them.

```bash
kinoko import [path...] [flags]
kinoko import --dir ./logs/
```

**Arguments:**

| Argument | Description |
|---|---|
| `[path...]` | One or more session log file paths |

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--config` | string | `""` | Config file path |
| `--library` | string | first configured | Library ID |
| `--dir` | string | `""` | Directory of log files to import (`.log`, `.txt`, `.json`) |

**Examples:**
```bash
kinoko import session.log
kinoko import a.log b.log c.log
kinoko import --dir ./logs/
kinoko import --dir ./logs/ --library my-team
```

**Behavior:**
- Files larger than 50 MB are skipped
- Provide file paths as arguments, `--dir`, or both

## `kinoko match`

Query the Kinoko server for skills matching a text query. Useful for testing injection relevance or exploring the skill library.

```bash
kinoko match <query-text> [flags]
kinoko match --file <context.txt> [flags]
```

**Arguments:**

| Argument | Description |
|---|---|
| `<query-text>` | Free-text query describing what you need (optional if `--file` is used) |

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--file` | string | `""` | Read query text from a file instead of the argument |
| `--limit` | int | `5` | Maximum number of results to return |
| `--min-score` | float | `0.5` | Minimum match score threshold (0.0–1.0) |
| `--api-url` | string | `$KINOKO_API_URL` or `http://127.0.0.1:23233` | Kinoko API URL |
| `--timeout` | duration | `30s` | Command timeout |

**How it works:**

1. Takes the query text (from argument or `--file`)
2. Sends it to the server's match endpoint
3. Server computes embeddings and finds nearest skills
4. Results are filtered by `--min-score` and limited by `--limit`
5. Prints ranked results with names, scores, and content previews

**Examples:**
```bash
kinoko match "I need to fix a database timeout in Go"
kinoko match --file context.txt --limit 10
kinoko match "memory leak" --min-score 0.7
kinoko match "deployment strategies" --api-url http://kinoko.internal:23233
```

**Output:**
```
─── Match Results (3) ───
  1. fix-db-connection-pool (score: 0.847)
     Configure connection pool timeouts and max idle connections...
  2. go-context-cancellation (score: 0.723)
     Use context.WithTimeout to propagate deadline...
  3. retry-with-backoff (score: 0.651)
     Implement exponential backoff for transient failures...
─────────────────────────
```

## `kinoko scan`

Scan files for embedded credentials, API keys, tokens, and other secrets.

```bash
kinoko scan [file] [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--dir` | string | `""` | Scan a directory recursively |
| `--stdin` | bool | `false` | Read content from stdin (for git hooks) |
| `--reject` | bool | `false` | Exit 1 on findings (for git hooks) |

**Examples:**
```bash
kinoko scan config.yaml          # Scan a single file
kinoko scan --dir ./skills       # Scan a directory recursively
kinoko scan --stdin              # Scan stdin (for git hooks)
echo "sk-abc123..." | kinoko scan --stdin --reject
```

**Behavior:**
- Uses the same 14-pattern credential scanner as the pre-receive hook
- High-confidence findings (≥0.7) are reported
- With `--reject`, exits with code 1 when credentials are found
- Stdin is limited to 10 MB

See [Security](/concepts/security/) for details on the scanning patterns.

## `kinoko index`

Index a skill repo into SQLite. Typically invoked by the post-receive hook, but can be run manually.

```bash
kinoko index [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--repo` | string | `$KINOKO_REPO` | Repo name (e.g. `local/fix-nplus1`) |
| `--rev` | string | `$KINOKO_REV` | Git revision (commit hash) |
| `--dsn` | string | `$KINOKO_STORAGE_DSN` | SQLite database path |

**Environment variables:**

| Variable | Purpose |
|---|---|
| `KINOKO_REPO` | Repo name (fallback for `--repo`) |
| `KINOKO_REV` | Git revision (fallback for `--rev`) |
| `KINOKO_STORAGE_DSN` | SQLite path (fallback for `--dsn`, default `~/.kinoko/kinoko.db`) |

**Example:**
```bash
kinoko index --repo local/fix-nplus1 --rev HEAD
```

## `kinoko pull`

Clone or update skill repositories from the connected Kinoko server.

```bash
kinoko pull [skill-repo] [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--all` | bool | `false` | Sync all cached skills |

**Examples:**
```bash
kinoko pull local/fix-nplus1     # Clone or update a specific skill
kinoko pull --all                # Sync all cached skills
```

**Behavior:**
- Requires a server connection (set up via `kinoko init --connect`)
- Skills are cached locally in `~/.kinoko/cache/`

## `kinoko queue`

Queue inspection and management subcommands.

```bash
kinoko queue stats     # Print queue depth and status counts
kinoko queue list      # List queued, pending, and errored sessions
kinoko queue retry <session-id>  # Requeue a failed session
kinoko queue flush     # Delete all queued sessions from the queue
```

**Subcommands:**

| Subcommand | Description |
|---|---|
| `stats` | Print queue depth and status counts |
| `list` | List queued, pending, and errored sessions |
| `retry <session-id>` | Requeue a failed session for re-extraction |
| `flush` | Delete all queued sessions from the queue |

## `kinoko rebuild`

Rebuild the SQLite cache from git repos. Scans the repos directory for bare git repositories containing SKILL.md files, parses and indexes each one into SQLite.

```bash
kinoko rebuild [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--clean` | bool | `false` | Drop skill tables before rebuilding |
| `--library` | string | `""` | Only rebuild repos under this library prefix |
| `--dsn` | string | `$KINOKO_STORAGE_DSN` | SQLite DSN |
| `--api-url` | string | `$KINOKO_API_URL` or `http://127.0.0.1:23233` | Kinoko API URL |
| `--data-dir` | string | `$SOFT_SERVE_DATA_PATH` or `~/.kinoko/data` | Soft Serve data directory |

**Examples:**
```bash
kinoko rebuild
kinoko rebuild --clean
kinoko rebuild --library my-team --clean
```

## `kinoko stats`

Query the database and print client pipeline metrics.

```bash
kinoko stats [flags]
```

**Flags:**

| Flag | Type | Default | Description |
|------|------|---------|-------------|
| `--config` | string | `""` | Config file path |

**Metrics reported:**

| Section | Metrics |
|---|---|
| Sessions | Total, extracted, rejected, errors, extraction yield |
| Stage Pass Rates | Per-stage passed/total/rate |
| Extraction Precision | Human review: reviewed, useful, precision % |
| Injection Metrics | Events, sessions with injection, injection rate, skill utilization |
| A/B Test Results | Treatment/control sessions, success rates, z-score, p-value, significance |
| Skills by Category | Count per category (foundational, tactical, contextual) |
| Quality Scores | Average composite score, average confidence |
| Decay Distribution | Bucketed: dead, low, medium, high, fresh |

## Exit Codes

| Code | Meaning |
|------|---------|
| 0 | Success |
| 1 | Error (config issues, startup failures, runtime errors, credential scan findings) |
| 2 | Rejected — extraction rejected or ingest rejected by critic (`extract`, `ingest`) |
| 3 | Extraction error (`extract` only) |
| 130 | Interrupted (Ctrl+C) |
