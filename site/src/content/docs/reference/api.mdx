Kinoko's HTTP API runs on port **23233** by default (configurable). All endpoints accept and return JSON.

## Endpoints at a Glance

| Method | Path | Purpose |
|---|---|---|
| `GET` | `/api/v1/health` | Server health check |
| `POST` | `/api/v1/discover` | Unified discovery (prompt/embedding/patterns) |
| `POST` | `/api/v1/embed` | Generate text embeddings |
| `POST` | `/api/v1/ingest` | Submit a session for extraction |
| `GET` | `/api/v1/skills/decay` | List skills by decay score |
| `PATCH` | `/api/v1/skills/{id}/decay` | Update skill decay score |

---

## `GET /api/v1/health`

Health check. Returns server status and skill count.

**Response:**
```json
{
  "status": "ok",
  "skills": 142
}
```

## `POST /api/v1/discover`

**Unified discovery endpoint** that handles all skill query patterns — by prompt, pre-computed embedding, patterns, or any combination. This consolidates what used to be separate search, match, and discovery endpoints.

**Request:**
```json
{
  "prompt": "database performance optimization", 
  "embedding": [0.023, -0.118, 0.042, "...384 floats"],
  "patterns": ["postgres", "performance"],
  "library_ids": ["default", "team-backend"],
  "min_quality": 0.7,
  "top_k": 10
}
```

| Field | Required | Default | Description |
|---|---|---|---|
| `prompt` | Conditional* | — | Natural language query (server will embed) |
| `embedding` | Conditional* | — | Pre-computed vector (skip server embedding) |
| `patterns` | Conditional* | — | Keyword patterns to filter by |
| `library_ids` | No | all | Limit to specific skill libraries |
| `min_quality` | No | `0.0` | Minimum quality threshold |
| `top_k` | No | `10` | Maximum results to return |

*At least one of `prompt`, `embedding`, or `patterns` must be provided.*

**Response:**
```json
{
  "skills": [
    {
      "repo": "default/postgres-query-optimization",
      "name": "postgres-query-optimization",
      "description": "",
      "score": 0.72,
      "clone_url": "ssh://localhost:23231/default/postgres-query-optimization"
    }
  ]
}
```

**Use Cases:**
- **Injection queries:** Client provides context as `prompt`, gets matching skills for session injection
- **Pre-computed embedding:** Client has embedding ready, skips server embedding step  
- **Pattern-based search:** Filter skills by specific keywords or tags
- **Library scoping:** Search within specific team/project libraries only

<Aside type="note">
If `prompt` is provided without `embedding`, the server will generate the embedding internally using the configured model (`bge-small-en-v1.5`).
</Aside>

## `POST /api/v1/embed`

Generate a vector embedding for text. Requires the [embedding engine](/reference/embedding-engine) to be configured.

**Request:**
```json
{
  "text": "How to fix CORS errors in Express.js"
}
```

**Response:**
```json
{
  "vector": [0.023, -0.118, 0.042, "...384 floats"],
  "model": "bge-small-en-v1.5",
  "dims": 384
}
```

**Errors:**
- `400` — missing or empty `text`
- `503` — embedding engine not available (built without `embedding` tag)

## `POST /api/v1/ingest`

Submit a session log for extraction processing. This is typically called by the git post-receive hook when skills are pushed to the server.

**Request:**
```json
{
  "session_id": "abc123",
  "log": "... session transcript ..."
}
```

**Response:**
```json
{"queued": true}
```

Returns `501 Not Implemented` if no enqueue function is configured (use `kinoko run` to process sessions instead).

<Aside type="note">
Client extraction now uses a local queue instead of pushing directly via this endpoint. The ingest endpoint is primarily used for server-side processing when an enqueue handler is configured.
</Aside>

## `GET /api/v1/skills/decay`

List skills ordered by decay score. Used by the server scheduler to identify skills that may need attention or removal.

**Response:**
```json
{
  "skills": [
    {
      "id": "outdated-react-patterns",
      "name": "outdated-react-patterns",
      "repo": "default", 
      "decay_score": 0.95,
      "last_used": "2026-01-15T10:30:00Z",
      "created": "2025-11-20T14:22:00Z"
    }
  ]
}
```

---

## `PATCH /api/v1/skills/{id}/decay`

Update the decay score for a specific skill. Used by the scheduler to manage skill freshness.

**Request:**
```json
{
  "decay_score": 0.3,
  "last_used": "2026-02-17T15:45:00Z"
}
```

**Response:** `200 OK` on successful update.

---

## Rate Limiting

The discover endpoint uses a semaphore limiting concurrent requests to 10. Other endpoints have a 1MB request body limit.

## Security

- Ingest request bodies are capped at 10 MB; embed bodies at 1 MB (`http.MaxBytesReader`)
- Input validation on discover endpoint (max 2048 embedding dims, max 50 patterns/library_ids)
- No authentication by default — designed for localhost/private network use
