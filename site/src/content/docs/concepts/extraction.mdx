## The Model

A gold miner doesn't analyze every grain of sand with a microscope. They start with a pan, washing away the obviously worthless material. What survives gets progressively finer treatment — magnetic separation, chemical assays — until only gold remains. Each stage is more precise *and* more expensive than the last.

## Pipeline Overview

Most agent sessions don't produce reusable knowledge. Running an expensive LLM critic on every session would be wasteful. Instead, Kinoko filters in three stages, rejecting early and cheap:

```
  Session
    │
    ▼
  ┌────────────┐  ~70%
  │  Stage 1   │──rejected──▶ ✗
  │  Metadata  │
  └─────┬──────┘
        │ ~30% pass
        ▼
  ┌────────────────┐  ~50%
  │    Stage 2     │──rejected──▶ ✗
  │ Novelty+Rubric │
  └─────┬──────────┘
        │ ~15% pass
        ▼
  ┌────────────┐  ~60%
  │  Stage 3   │──rejected──▶ ✗
  │ LLM Critic │
  └─────┬──────┘
        │ ~6% pass
        ▼
  ┌───────────────┐  duplicate
  │ Novelty Check │──────────▶ ✗
  └─────┬─────────┘
        │ novel
        ▼
    git push
```

## Stage 1 — Metadata Pre-filter

**Cost:** near-zero. **Purpose:** eliminate the obviously worthless.

Fast checks against session metadata — no content analysis needed:

- **Minimum duration** — sessions too short to contain meaningful work
- **Minimum tool calls** — sessions where the agent didn't actually do anything
- **Maximum error rate** — sessions that were mostly failures

This stage rejects the bulk of sessions instantly. A session that lasted 3 seconds with zero tool calls isn't worth analyzing further.

## Stage 2 — LLM Rubric Scoring

**Cost:** moderate (one LLM call). **Purpose:** filter low-quality sessions before expensive extraction.

An LLM call scores the session across 7 quality dimensions: problem specificity, solution completeness, context portability, reasoning transparency, technical accuracy, verification evidence, and innovation level. Sessions below the minimum viable threshold are rejected.

Stage 2 also classifies the session's category (foundational, tactical, contextual) and identifies matching patterns from the taxonomy.

## Stage 3 — Combined Critic + SKILL.md Generation

**Cost:** high (full LLM call). **Purpose:** final quality gate + knowledge synthesis.

This is the expensive stage — and it only runs on the ~15% of sessions that survived Stages 1 and 2. A single LLM call does both:

1. **Evaluate** the session with a detailed critic rubric
2. **Generate** the SKILL.md content if the verdict is "extract"

### Quality Controls

Three mechanisms from the [brief-010 prompt tuning](https://github.com/kinshitai/kinoko) prevent project-specific false positives:

- **"Different Project" preamble** — the prompt tells the LLM: *"You are extracting skills for a DIFFERENT agent working on a DIFFERENT project. That agent has never heard of this project."* This reframes the evaluation from "is this useful?" to "is this transferable?"

- **Substitution test** — *"Replace all project-specific names with generic placeholders. Does the remaining skill still contain non-obvious, actionable knowledge?"* If context portability scores below 3/5, the skill is rejected regardless of other scores.

- **Hard reject triggers** — automatic rejection if:
  - ≥3 proper nouns from the source project are essential to the solution
  - Removing project context reduces the skill to fewer than 2 sentences
  - The skill documents *what* was built rather than reusable *how/why*
  - A competent developer would say "that's just how you'd do it"

### Output

If verdict is "extract", Stage 3 produces a complete SKILL.md with front matter, problem statement, solution steps, reasoning, and pitfalls — ready to push to git.

## After Extraction

Skills that pass Stage 3 go through:

1. **Credential scanning** — 14 secret patterns checked, findings above confidence threshold redacted
2. **Novelty check** — final server-side duplicate check
3. **Git push** — skill is pushed as its own repo to the server
4. **Post-receive hook** — server parses the SKILL.md, embeds it, indexes into SQLite

## Cost Model

The staged approach keeps costs practical:

| Stage | Cost per session | % of sessions reaching it |
|-------|-----------------|--------------------------|
| Stage 1 | ~$0 | 100% |
| Stage 2 | ~$0.001 | ~30% |
| Stage 3 | ~$0.04 | ~15% |

**Stage 3 accounts for ~98% of total extraction cost.** Overall: roughly **$6 per 1,000 sessions** processed. The staged filtering means you only pay the expensive price for sessions likely to yield real knowledge.

## Resilience

The pipeline includes built-in resilience for LLM calls:

- **Retry logic** with exponential backoff for transient failures
- **Circuit breaker** — 5 consecutive failures triggers a 5-minute cooldown
- **Fail-open novelty** — server unavailability doesn't block extraction

## For Contributors

When working on extraction, think about which stage a filter belongs in:

- Can it be evaluated from metadata alone? → **Stage 1**
- Does it need content analysis but not judgment? → **Stage 2**
- Does it require nuanced evaluation? → **Stage 3**

Moving a filter to an earlier stage saves cost. Moving it later improves precision.
